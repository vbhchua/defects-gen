Why this build

Motivation and takeaways

The question

  • How easy is it for a solo data scientist to get this up and running end-to-end — with agents as your AI-platform partner?

Takeaways

  • Pretty good — it ran end-to-end without first learning OSMO, Kubernetes, or kind cluster setup; the agent carried the platform work.
  • The sim-to-real gap is still an open problem — and how wide it is depends on the domain.
  • Scoped down, it's already useful — glass defects, where only the defect is synthetic and the background is a real photo, transferred best.

Sanity check on real data

Impact and Insights: Sim-to-real transfer

Metal — 0.39 real mAP@50 (vs 0.93 synthetic)

386 real Magnetic Tile photos. Recall holds (AR ~0.6) but precision and box quality drop. Crack transfers worst (0.07–0.10 AP@50) — real hairline cracks differ from the synthetic renders. Both background and defect are synthetic here.

Glass — 0.56 real mAP@50 (vs 0.97 synthetic)

300 real mobile-screen photos. Oil transfers well: 0.949 AP@50 real. Scratch and stain are detected but boxed loosely — real elongated scratches differ from the synthetic point-defect shapes.

Glass model on real photos — GT vs predictionstain found (0.79) · scratches found but boxed loosely · one real oil smudge misread as stain
Why glass transfers better: its synthetic images use real phone photos as backgrounds — only the defect is synthetic; metal synthesizes both. Next steps: mix real images into training, add base backgrounds, diversify defect geometry.
The hard parts

What made it hard — four failures, four root causes

› The "out of memory" that wasn't exit 2137

  • Felt like: every setup run died "out of memory" after ~20 minutes — with memory to spare on the host.
  • Why: pods got no S3 address settings, so the client fell back to a hostname that can't resolve — and retried silently against a 24-hour window, leaking 13 MB/s until the pod was killed.
  • Fix: inject the S3 endpoint + path-style env via POD_TEMPLATE. A memory symptom, an addressing bug.

› Everything broke overnight exit 2014

  • Felt like: after a host reboot, every workflow failed within seconds — including yesterday's working ones.
  • Why: the default store kept its data on /tmp, wiped by the reboot — and the bucket only ever existed via an install-time hook.
  • Fix: durable, host-mounted storage (MinIO on the 1 TB volume).

› The token that passed every check KeyError

  • Felt like: the HF token read gated repos fine from the laptop, then crashed inside the pod.
  • Why: an OAuth token isn't a user access token — the in-pod login needs the real thing; and OSMO snapshots credentials at submit time.
  • Fix: mint a user access token, and fix credentials before submitting.

› The upload that went backwards NoSuchUpload

  • Felt like: an 80 GB upload crawling to 40%, then regressing — indefinitely.
  • Why: the default store locks itself every 10 s to persist; stalled clients retry and race its own part-cleanup.
  • Fix: replace with MinIO — the same upload completed at 63.6 MB/s with zero errors.
Full analysis with logs and timelines in PROGRESS.md · each failure signature is documented for quick diagnosis.
Closing the loop

Synthetic data → object detector, all local

APACHE 2.0
🤖

RF-DETR Large

Roboflow's real-time DETR (ICLR 2026): DINOv2 backbone, 33.9 M params, 704×704. Largest Apache-2.0-licensed variant — XL/2XL are PML 1.0.

FREE LABELS
🏷️

Boxes from masks

DIG emits pixel masks, not boxes. Connected components on original_mask → COCO bboxes; class from the filename. No manual annotation needed.

NO LEAKAGE
✂️

Group splits

All images derive from just 20 base backgrounds (20–45 variants each). Splits are grouped by base image — test backgrounds are never seen in training.

2 DATASETS
🗂️

Metal & glass

Metal: 629 imgs, 5 classes (Blowhole, Break, Crack, Fray, Uneven). Glass: 900 phone-screen imgs, 3 classes (oil, scratch, stain), perfectly balanced.

~10 MIN
⚡

✓ Glass — 0.97 mAP@50

No cross-class confusion: oil 1.00 AP@50 (45/45), scratch & stain 0.95. All 9 errors are missed small defects or loose boxes on faint thin scratches — none are a wrong class.

✗ Metal — MT_Break lags

Blowhole/Fray/Uneven at 0.92–0.93 AP@50, but MT_Break 0.67 — low-contrast smudges: 8/26 missed, most false positives. It also had only 2 valid-split samples; more Break data is the fix.

The small valid→test gap (metal 0.73→0.69, glass 0.76→0.78 mAP@50:95) indicates the detectors generalize across held-out synthetic backgrounds rather than memorizing them — transfer to real photos is measured separately on the sim-to-real slide.
Where it goes

Next steps

1
Scale the metal dataset
Re-submit with larger num_sdg — the fixed input pull amortizes, so big batches are cheap per image.
2
Close the sim-to-real gap
Real-data validation is done (0.39 / 0.56 real mAP@50). Next: mix a small set of real images into training, generate from new base backgrounds, and fix thin-defect geometry (cracks, elongated scratches).
3
Finetune from scratch
Train a custom checkpoint on our own defect data (use_pretrained_checkpoint=false) — the single GPU holds the job for its duration.
4
PCBA Day 0 — full synthetic pipeline
USD render → image-edit → AnomalyGen. Needs an image-edit endpoint decision first: the pool has 1 GPU, so a local NIM can't run beside a DIG job.
5
Production hardening
Real IdP instead of the no-auth gateway, external S3, multi-GPU pool — the values files are the only thing that changes.

✗ localstack (chart default)

Same 80.6 GB upload: failed at 40% after 26 min — NoSuchUpload on every multi-GB file, progress regressing, stopped by the 1 h timeout. Bucket also lost on every reboot.

✓ MinIO (this repo)

Identical upload: completed in 22:40 at 63.6 MB/s average (peaks 305 MB/s), zero multipart errors, data durable on the host's 1 TB volume across reboots.

Scaling: the 22-min input pull is a fixed cost — 30 images cost ~16 s/image end-to-end, but ~300 images ≈ 1.7 h, not 10× the 31-min run. Larger batches amortize the fixed cost.
