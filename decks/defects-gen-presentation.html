The findings

Four failures, four root causes

› The 18-minute OOM spiral exit 2137

Task pods get no S3 env from the chart → boto3 falls back to virtual-hosted DNS → unresolvable → ctrl leaks 13 MB/s against a 24 h retry window until cgroup OOM. Fix: endpoint + path-style env in POD_TEMPLATE, both containers.

› The vanishing bucket exit 2014

The chart's localstack data dir was bind-mounted to host /tmp — wiped every reboot, and the bucket only exists via an install-time hook. Every workflow died in 3 s; even the logs API returned NoSuchBucket.

› The token that half-works KeyError

An HF OAuth token reads gated repos (probes pass!) but crashes in-pod hf auth login — no accessToken in whoami. Bonus trap: OSMO snapshots credentials at submit time — fix creds before submitting.

› Multipart roulette NoSuchUpload

localstack-persist write-locks the server every 10 s; under an 80 GB upload, stalled clients retry CompleteMultipartUpload and race its part-cleanup. Upload regressed 40%→34% forever. Fix: replace with MinIO — zero errors, first try.

Full forensics with logs and timelines: PROGRESS.md · every signature is now a 30-second diagnosis.

✗ localstack (chart default)

Same 80.6 GB upload: died at 40% after 26 min — NoSuchUpload on every multi-GB file, progress moving backwards, killed by the 1 h timeout. Plus: bucket lost on every reboot.

✓ MinIO (this repo)

Identical upload: completed in 22:40 at 63.6 MB/s average (peaks 305 MB/s), zero multipart errors, data durable on the host's 1 TB volume across reboots.

📈 Scaling: the 22-min input pull is a fixed cost — 30 images cost ~16 s/image end-to-end, but ~300 images ≈ 1.7 h, not 10× the 31-min run. Batch big.
