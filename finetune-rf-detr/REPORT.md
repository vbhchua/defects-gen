# RF-DETR Large Finetuning Report — DIG Metal-Surface Defect Detection

**Date:** 2026-07-12 · **Hardware:** 1× NVIDIA RTX PRO 6000 Blackwell (96 GB) · **Everything ran locally on this machine.**

## 1. Objective

Finetune an object-detection model on the synthetic metal-surface defect images
generated by the DIG (Defect Image Generation) pipeline, with proper
train/valid/test methodology, and quantify its performance.

**Model choice:** [RF-DETR](https://github.com/roboflow/rf-detr) **Large**
(`RFDETRLarge`, 33.9 M params, 704×704, DINOv2 backbone). The `rfdetr` package
and the Large weights are **Apache 2.0**; only the XL/2XL "Plus" models carry
the restrictive PML 1.0 license, so Large is the biggest permissively-licensed
variant.

**Dataset choice:** the metal-surface dataset (the project's primary use case).
A phone-screen dataset (900 imgs, oil/scratch/stain) also exists under
`data/texture_defect_gen_day1_manual_roi-6771352a` and is a natural follow-up.

## 2. Dataset construction

Sources (DIG Day-1 manual-ROI runs):

| Run | Images | Notes |
|---|---|---|
| `…-c2f5afb4` | 600 | main metal run, 120 per class |
| `…-51f57bdd` | 30 | small pilot run, 6 per class |

- **Images:** `inference/reconstructed_image/*.png` (grayscale-like PNGs, 237×272 – 529×315 px).
- **Labels:** no bbox annotations existed. Boxes were **derived from the pixel
  masks** (`inference/original_mask`) via 8-connected components (min area 9 px²);
  class parsed from the filename. The pipeline's own `annotated_image` overlays
  ("Mask BBOX") confirm this is the intended ground truth.
- **Dedup:** images hashed (MD5) across runs; 1 exact duplicate removed → **629 unique images, 674 boxes** (96 % have a single defect; max 6).
- Box sizes: median 66×65 px, min 2×3, max 521×384 (MT_Uneven regions are legitimately large).

### Leakage-free split (the important part)

The 629 images derive from only **20 unique base clean images** (20–42 synthetic
variants per base, per `SDG_result.csv`). A naive random split would put
near-identical backgrounds in both train and test and inflate metrics. Splits
are therefore **grouped by base clean image** (greedy size-balanced assignment,
seed 42), verified so all 5 classes appear in every split:

| Split | Base groups | Images | Boxes | Blowhole / Break / Crack / Fray / Uneven |
|---|---|---|---|---|
| train | 13 | 439 | 475 | 86 / 101 / 99 / 67 / 86 |
| valid | 3 | 85 | 89 | 20 / 2 / 13 / 30 / 20 |
| test  | 4 | 105 | 110 | 19 / 23 / 14 / 29 / 20 |

Caveat: with only 20 groups, per-class balance inside small splits can't be
fully controlled (valid has just 2 MT_Break images). Test is reasonably
balanced. COCO-format output: `dataset/{train,valid,test}/_annotations.coco.json`.

## 3. Training

`train.py`: `RFDETRLarge` from COCO-pretrained weights, square-resize 704×704,
`batch_size=16`, `grad_accum_steps=1` (effective batch 16 as recommended),
default LR schedule, EMA on, `early_stopping=True` (patience 10), max 40 epochs.

- Stopped at **epoch 35** (best val at epoch 25). Wall time **≈ 9.5 min**, peak ~26 GB VRAM.
- Best validation mAP@50:95: **0.732** (regular, epoch 25); best EMA: 0.722 (epoch 33).
- Curves: `eval/training_curves.png` — loss decreases smoothly; val mAP plateaus ≈ epoch 25 with mild oscillation typical of a small val set.

|---|---|---|

| mAP@50 | 0.890 | **0.851** |

Per-class (test):

| Class | AP@50:95 | AP@50 | AR | GT boxes |
|---|---|---|---|---|
| MT_Blowhole | 0.723 | 0.916 | 0.858 | 19 |

The small valid→test gap (0.73 → 0.69) indicates the group split held and the
model generalizes across backgrounds rather than memorizing them.

### Error analysis (confusion at conf 0.5, IoU 0.5 — `eval/confusion_matrix_test.png`)

- **MT_Break is the weak class**: 18/26 detected correctly, 8 missed, and it
  also draws the most false positives (9). Breaks are low-contrast smudge-like
  regions that resemble background shading; the valid split had almost no
  Breaks (n=2), so model selection couldn't optimize for it.
- Minor confusions: 3 Blowhole→Fray, 1 Blowhole→Crack, 2 Fray→Crack — all
  visually plausible (small dark specks vs. thin dark streaks).
- MT_Uneven is essentially solved (AR 0.96) — large distinctive regions.
- Small objects (< 32² px) remain the hardest (AP 0.52), typical for detection.

### Inference speed

**11.2 ms/image** (~89 FPS) end-to-end `model.predict()` after
`optimize_for_inference()`, single image at a time, on this GPU.

Qualitative GT (left) vs prediction (right) pairs: `eval/predictions_test.png`.

## 5. Conclusions & recommendations

- RF-DETR Large finetunes quickly and well on this synthetic dataset:
  **0.85 mAP@50 / 0.69 mAP@50:95 on a background-disjoint test set**, at
  real-time speed, with the full pipeline (data prep → train → eval) local.
- **Data, not model, is the current bottleneck**: only 20 base backgrounds and
  ~630 images. Generating more variants from *new* base images would improve
  both the model and the reliability of evaluation (test CIs are wide at
  ~110 boxes).
- Prioritize more **MT_Break** examples (and Breaks in the valid split) — it's
  the weakest class by a wide margin.
- Evaluation on *real* (non-synthetic) defect images is the key missing
  validation before any deployment claim: this report measures
  synthetic→synthetic generalization only.
- Follow-up: same pipeline on the phone-screen run (900 imgs, 3 classes).

## 6. Reproduce

```bash
cd /home/ubuntu/defects-gen/finetune-rf-detr
.venv/bin/python build_dataset.py   # dataset/ + manifest.json
.venv/bin/python train.py           # output/ checkpoints (~10 min)
.venv/bin/python evaluate.py        # eval/metrics.json + figures
```

Environment: Python 3.12 venv, `torch 2.13.0+cu129` (Blackwell sm_120 needs a
cu128+ build — install torch from the cu129 index *before* `rfdetr`),
`rfdetr[train,loggers]`, `supervision 0.29`, `pycocotools`.
Note: `model.predict()` returns **0-indexed** class ids; COCO category ids in
the annotations are 1-indexed — map with `+1` when scoring against COCOeval.
